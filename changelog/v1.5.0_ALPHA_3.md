# 1.5.0_ALPHA_3 — 2026-05-06

Comprehensive security and stability release — 113 issues fixed across permissions, async networking, concurrency, block-entity contracts, UI precision, and multi-server resilience. Server administrators should review permission configurations as several bypasses are now correctly enforced.

## Security

| Tag | Change |
|---:|---|
| ![bugfix] | <details><summary>Permission bitmask bypass in deposit, withdraw, and admin checks</summary><br>`BankPermission.ordinal()` returns 0/1/2 but bitmask values are 1/2/4. All three call sites now use `getValue()` so DEPOSIT/WITHDRAW/MANAGE checks enforce the correct bit.</details> |
| ![bugfix] | `/banksystem deop` (self and user variants) granted admin instead of revoking — now correctly passes `false` |
| ![bugfix] | `DeleteBankAccountAsync` permission gate used OR instead of AND, blocking deletion when the player was non-admin OR lacked MANAGE — now admin OR MANAGE allows deletion |
| ![bugfix] | <details><summary>BankUploadBlockEntity ownership bypass</summary><br>`handlePacket` applied `dropIfNotBankable` and `bankAccountNumber` settings before the ownership check, letting any sender redirect items to a chosen account. Settings now applied only inside the ownership guard.</details> |
| ![bugfix] | <details><summary>UpdateBankAccountRequest accepted arbitrary permission values</summary><br>Clients with MANAGE could set any permission integer. Values are now masked to the valid bitmask, and the bank owner's permissions can no longer be modified by other users.</details> |
| ![bugfix] | Client-callable `SetAccountNameAsync` and `SetAccountIconAsync` had no permission check — now requires MANAGE permission |
| ![bugfix] | <details><summary>Integer overflow in WithdrawMoneyPacket allowed item duplication</summary><br>`requestedAmount * itemValue` could overflow `long`, letting crafted packets withdraw a tiny amount but receive huge stacks. Added pre-multiplication overflow guard. Same overflow guard added to `putRemainingItemsBack` redeposit path.</details> |
| ![bugfix] | `BankTerminalBlockDataRequest` cast block entity without `instanceof` — malicious client with wrong BlockPos could crash the server |
| ![bugfix] | `WithdrawItemsFromBankRequest` returned requested items on every error path, signaling success when none occurred — now returns empty map on rejection |
| ![bugfix] | `DepositItemsInBankRequest` accepted negative amounts — now clamped to zero |
| ![bugfix] | Bank block GUIs opened even on disconnected slave servers — `canInteractWithBankSystem()` return value was discarded in upload, download, and terminal blocks |
| ![bugfix] | <details><summary>Item duplication via async race in upload and download block entities</summary><br>Upload block: inventory slot was read on tick N, async deposit started, slot only cleared in callback — tick N+1 could read and deposit the same items again. Fix: clear slot optimistically before async chain, restore on failure.<br><br>Download block: `currentlyReceiving` flag reset synchronously after spawning inner async futures — next tick could start duplicate withdrawals. Fix: reset flag only after all inner futures complete via `CompletableFuture.allOf`.</details> |
| ![bugfix] | <details><summary>TOCTOU withdraw race in terminal block allowed overdraft</summary><br>`sendToBlock` read balance, computed withdraw amount, added items to inventory, then withdrew — another operation could drain funds between the balance read and withdrawal. Replaced with lock-withdraw-unlock pattern matching `BankDownloadBlockEntity`.</details> |
| ![hardening] | 8-block range check added to all four block-entity packet handlers, preventing interaction with entities anywhere in the loaded world |
| ![hardening] | <details><summary>Untrusted slave servers blocked from mutating operations</summary><br>`SetAccountNameAsync` and `SetAccountIconAsync` removed from the untrusted-slave whitelist in `AsyncBankAccount` — untrusted slaves could rename accounts or change icons without permission. `DeleteBankAccountAsync`, `AllowItemIDAsync`, and `DisallowItemIDAsync` removed from the untrusted-slave whitelist in `AsyncBankManager` — untrusted slaves could delete accounts or modify the bankable-item list.</details> |
| ![hardening] | `SetBanksystemAdminModeAsync` handler on the master now verifies the requester is already an admin before allowing privilege changes — previously any trusted slave could promote arbitrary players |
| ![hardening] | `BankUploadBlockEntity.sendInventoryToBank()` async callbacks now run on the server thread — previously ran on ForkJoinPool, risking item duplication and crashes from off-thread world/inventory mutation |

## Performance

| Tag | Change |
|---:|---|
| ![bugfix] | <details><summary>Netty buffer leak and oversized wire payloads in async operations</summary><br>`AsyncFunctionInputData.of` and `AsyncFunctionOutputData.of` allocated `Unpooled.buffer()` and never released them — every async call leaked the buffer. Both now wrapped in try/finally with `release()`.<br><br>Additionally, `buf.array()` returned the full 256-byte backing array including unused capacity. Replaced with `ByteBufUtil.getBytes(buf)` to send only the readable region.</details> |
| ![bugfix] | BankAccountChangeStream listener leak — each subscribe/unsubscribe created a new method reference instance, preventing unsubscribe from matching the original listener. Now uses a stable `Consumer` field reference |
| ![bugfix] | ByteBuf leak in `decodeParams()` and `decodeResult()` — `RegistryFriendlyByteBuf` from `Unpooled.wrappedBuffer()` was never released; now wrapped in try/finally |
