# Changelog - v1.5.0_ALPHA_3

**Status:** In Development
**Release Date:** TBD
**Minecraft Version:** 1.21.1
**Platforms:** Fabric, NeoForge, Quilt

## Overview

Comprehensive bug-fix release addressing 46 issues identified through a full code analysis pass: critical permission/security vulnerabilities, async forwarding failures, hanging futures, networking errors, block-entity contract violations, arithmetic precision, and concurrency hazards. Singleplayer and dedicated-server in-game tests pass.

---

## Security

- **Fixed** - Permission check bitmask bypass: `BankPermission.ordinal()` returns enum positions (0,1,2) but the bitmask values are 1,2,4. Replaced `ordinal()` with `getValue()` in `DepositItemsInBankRequest`, `WithdrawItemsFromBankRequest`, and `AsyncBankManager` so DEPOSIT/WITHDRAW/MANAGE checks now test the correct bit.
- **Fixed** - `/banksystem deop` (self and user variants) was passing `true` and **granting** admin privileges instead of revoking them. Now correctly passes `false`.
- **Fixed** - `DeleteBankAccountAsync` permission gate used `||` (OR) which blocked deletion if the player was non-admin OR lacked MANAGE — should have been `&&` so admin OR MANAGE allows deletion.
- **Fixed** - `BankUploadBlockEntity.handlePacket` set `dropIfNotBankable` and `bankAccountNumber` **before** the ownership check, letting any sender redirect items to a chosen account. Settings are now applied only inside the ownership check.
- **Fixed** - `UpdateBankAccountRequest` accepted arbitrary permission integers from clients with MANAGE. Permission values are now masked to the valid bitmask (`BankPermission.getAllPermissions()`), and the bank owner's permissions can no longer be modified by other users.
- **Fixed** - Client-callable `SetAccountNameAsync` and `SetAccountIconAsync` had no permission check. The async master handler now requires the sender to have MANAGE permission on the account.
- **Fixed** - Integer overflow in `WithdrawMoneyPacket`: `requestedAmount * itemValue` could overflow `long` and let crafted packets withdraw a tiny amount but receive huge stacks. Added a pre-multiplication overflow guard (`requestedAmount > Long.MAX_VALUE / itemValue`).
- **Fixed** - `BankTerminalBlockDataRequest` cast the block entity without `instanceof`, allowing a malicious client with a wrong BlockPos to crash the server with `ClassCastException`. Replaced with a pattern-instanceof check.

## Fixed

### Async forwarding & networking

- `isSlaveServerTrustedAsync` was sending `FunctionType.IsBanksystemAdminAsync` (UUID input), so slave-trust validation was completely broken in multi-server mode. Now sends `IsSlaveServerTrustedAsync`.
- `GetMinimalDataAsync` codec was registered as `BankManagerData.STREAM_CODEC` but the handler returns `BankData`, causing serialization failure. Now uses `BankData.STREAM_CODEC`.
- `GetBankAccountNrByNameAsync` handler returned `IServerBankAccount` while the codec expected `int` — caused `ClassCastException`. Handler now extracts `getAccountNumber()` (or `INVALID_ACCOUNT_NUMBER` when no account is found).
- Four async lookups (`getPersonalBankAccountAsync(UUID)`, `getPersonalBankAccountAsync(String)`, `getPersonalBankAsync(UUID, ItemID)`, `getPersonalBankAsync(String, ItemID)`) never completed their futures when the account number was `<= 0`. Callers awaiting the future hung forever; now `complete(null)` in the negative branch.
- `bank_create_async` ran `handleResponse(...)` which casts results to `Boolean`, but `Bank_create` returns `Integer` (account number) — threw `ClassCastException`. Replaced with a custom completion handler that interprets the integer result.
- `removeAllBanksAsync` sent `FunctionType.RemoveEmptyBanksAsync` instead of `RemoveAllBanksAsync` — wrong server-side behavior.
- `WithdrawItemsFromBankRequest` returned `input.items` (the full requested amounts) on every error path, signaling success when none happened — could let callers create items from nothing. Now returns an empty map on rejection, and uses `getBank()` instead of `getOrCreateBank()` so empty bank entries are no longer created during withdrawal.
- `DepositItemsInBankRequest` did not validate negative deposit amounts — clamped to 0.

### Banking logic

- `withdrawLockedAsync` called `withdraw()` instead of `withdrawLocked()`, deducting from the wrong balance pool.
- `hasUserAsync(UUID)` called `hasAnyUser()` (any user at all) instead of `hasUser(userUUID)` for the specific user.
- `hasBankAsync(ItemID)` called `hasAnyBank()` instead of `hasBank(itemID)`.
- `toStringNoOwnerAsync` returned `toJsonString()` instead of `toStringNoOwner()`.
- `getNormalizedTotalBalanceAsync` and `getFormattedTotalBalanceAsync` used only `lockedBalance` instead of `balance + lockedBalance`.
- `createBankAccountAsync` was missing a `return` statement when the account-creation path failed, silently falling through to subsequent setup on a `null` account.
- `getOrCreatePersonalBankAccountAsync` checked the `CompletableFuture` reference for null (it never is) instead of the value — fallback creation never ran. Now delegates to `getOrCreatePersonalBankAccount_internal`.
- `getNotRemovableItems` accidentally returned `INITIAL_BLACKLIST_ITEMS` instead of `INITIAL_NOT_REMOVABLE_ITEMS`.
- `getOrCreatePersonalBankAccount` passed `INVALID_ACCOUNT_NUMBER` as the starting balance argument (semantically wrong, even though the value happened to be 0). Replaced with literal `0`.

### Arithmetic & precision

- `convertToRawAmountStatic("10.5")` produced `1005` instead of `1050` because the fractional part `"5"` was parsed as integer `5` rather than `50` hundredths. Fractional digits are now padded/truncated to the scale factor's decimal places before parsing.
- `convertToRealAmountStatic` declared `double` return but used `(float)` casts, losing precision past ~7 significant digits. Replaced with `double` casts.
- `getNormalizedAmountStatic` could divide by zero (`Math.log10(0) = -Infinity`), throw `StringIndexOutOfBoundsException`, and produce `NaN` for negative input. Now short-circuits on `amount <= 0`, handles the `wholeUnits == 0` case explicitly, and bounds-checks the substring slice.

### Concurrency

- `ServerBankManager.removeUser` mutated `bankAccounts` (via `deleteBankAccount`) while iterating its `entrySet()` — collected target account numbers into a list first, then deleted after iteration.
- `ServerBankAccount.unsubscribeBankChanges` removed listeners while iterating the listener list. Replaced with a direct `changeListeners.remove(callback)`.
- `BankAccountChangeStream.nextPacket` was written by the change-callback thread and read by the server tick thread without synchronization. Marked `volatile` so the tick thread sees updates and they aren't reordered.
- `BankSystemMod.init()` and `getAPI()` had non-synchronized lazy initialization that could create duplicate `BankSystemModBackend` instances. `init()` is now `synchronized`, the `backend` field is `volatile`, and `getAPI()` delegates to `init()`.

### Minecraft block-entity contracts

- `TerminalInventory.getItem`, `removeItem`, `removeItemNoUpdate` returned `null` for out-of-bounds slots while annotated `@NotNull`. The `Container` interface contract requires `ItemStack.EMPTY`; returning `null` causes NPEs in shift-clicks and hopper interaction.
- `TerminalInventory.load()` always returned `false` even on successful load — broke the `ServerSaveable` contract.
- `TerminalInventory.getItemCount` checked `if(itemID == null)` (the method parameter, never null) instead of the local `_itemID`. Fixed and tightened with `_itemID.isValid()`.
- `TerminalInventory.getFreeSpace` hardcoded `64` for empty slots; now uses the queried amount as an upper bound.
- `BankTerminalBlockEntity` called `ItemID.getOrRegisterFromItemStackClientSide` from server-side code paths (`getItemCount` variants). Replaced with `getOrRegisterFromItemStackServerSide_direct`.
- `BankTerminalBlock.onRemove` always dropped items, even when the new state was the same block (e.g., property changes). Added the `!state.is(newState.getBlock())` guard used by the upload/download blocks.
- `BankDownloadBlock.useWithoutItem` opened the menu inside a `CompletableFuture.thenAccept` callback, off the server main thread. Menu opening is now scheduled onto the main thread via `sPlayer.server.execute(...)`.
- `MoneyStockpileBlockEntity.dropItems` set the dropped stack count to `data.getAmount()`, which can exceed `maxStackSize` and produce invalid stacks. Now splits into multiple `ItemEntity` drops, each capped at the item's `maxStackSize`.

### Data persistence & lifecycle

- `BankSystemDataHandler.save_bank` continued to write to disk even when `bankManager` was null, overwriting valid saved data with an empty map. Now returns `false` early when `bankManager` is null.

### Commands

- `money_remove_user` compared `bank.getBalance()` (raw cents) against the supplied `amount` (real units) — any small balance passed the check. Now compares `bank.getRealBalance()` to the real `amount`.

### Account-data setters

- `setAccountName` no longer rewrites empty input to `""` (the empty-string fallback was redundant and caused a double-write); only the null case is normalized.
- `setAccountIcon` used `!=` for ItemID change detection, which fails for equal-by-value but distinct instances. Switched to `Objects.equals(...)`.

## Testing

### Fixed

- `ArithmeticTests.testConvertToRawAmountSingleDecimal` was asserting the buggy result (`1005`) as documentation of the original behavior. After the parser fix, it now asserts the correct result `1050`.
- `AsyncForwardingTests.testGetMinimalDataAsyncCodecIsBankData` compared codec object identity (`==`), but `AsyncFunctionDataCodecs` wraps every codec via `ExtraCodecUtils.nullable()`, so the wrapped codec is never identity-equal to the original. Replaced with a round-trip encode/decode of a `BankData` instance through the registered codec — verifies behavior independent of wrapping.

## Migration Notes

### For Server Administrators

- Permission system fixes change which actions are actually allowed: review existing permission entries, since previously some checks that appeared to pass (because `ordinal()` produced `0` and the bitmask AND was always satisfied) now correctly enforce permissions.
- `/banksystem deop` now actually revokes admin. If you previously used it expecting the old (wrong) behavior, switch to `/banksystem op` for granting.
- Multi-server slave-trust validation is now functional; ensure trusted-slave lists are configured correctly on the master.

### For Mod Developers

- Async lookup methods that previously hung on missing accounts now complete with `null`. If you were relying on the hang as a signal, switch to checking the result.
- `IServerBankAccount` from `getBankAccountNrByNameAsync` is no longer leaked through the codec — the function now returns the account number as advertised.
