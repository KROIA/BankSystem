# 2.0.3 — In Development

Fixes bank item identity for mods with time-varying item data (e.g. TerraFirmaCraft food decay): banked items no longer spoil, duplicate entries no longer appear, and spoiled items can no longer be laundered into fresh ones.

## Security

| Tag | Change |
|---:|---|
| ![bugfix] | <details><summary>Bank Upload Block duplicated damaged/enchanted items when "drop if not bankable" was enabled</summary><br>When the upload block encountered a damaged or enchanted stack with the drop-if-not-bankable option on, the stack was dropped on the ground **and** deposited into the bank account — duplicating the items. The drop branch now correctly skips the deposit.</details> |
| ![hardening] | <details><summary>Deposit equivalence gate — spoiled items can no longer be laundered into fresh ones</summary><br>With volatile components ignored for item identity, spoiled and fresh food share one bank entry — depositing a spoiled item and withdrawing a fresh one would have been possible. Now an item may only be deposited if it is equivalent to what the bank would hand back: the deposit check compares the incoming stack against a fresh withdrawal-equivalent stack using the game's own component-aware stack equality, so the source mod's rules (e.g. TFC's decay-stacking window) decide acceptability — no mod-specific code in BankSystem. Rejected deposits show a clear message to the player instead of failing silently (translated in all supported languages). Applies to Bank Terminal deposits and Bank Upload Block automation.<br><br>**Note for TFC packs:** food older than TFC's stack-merge window is rejected even if still edible — consistent with TFC's own rule that such food would not stack with fresh food in a chest. The window can be widened in TFC's config.</details> |
| ![hardening] | <details><summary>Dropped ItemID shorts can no longer be reassigned to an unrelated new item</summary><br>When a stored item template failed to resolve at world load (mod removed, corrupt NBT), the entry was silently dropped and its 16-bit short was free to be handed out again — silently rebinding downstream data (bank balances, StockMarket markets, plugin caches) to a different item. ItemID shorts are now allocated from a persisted monotonic counter that only ever moves forward, so a dropped short remains reserved for the lifetime of the world. Drop sites now log a `WARN` with the short and cached name; the counter refuses to wrap and returns an invalid ID on overflow. Legacy worlds migrate automatically on first load.</details> |

## API

| Tag | Change |
|---:|---|
| ![feature] | <details><summary>Volatile item components — datapack tag + config for pack authors</summary><br>Component types listed in the new data-component-type tag <code>banksystem:volatile_item_components</code> (shipped with optional entries <code>tfc:food</code>, <code>tfc:heat</code>) or in the new server config list <code>Bank.ADDITIONAL_VOLATILE_COMPONENTS</code> are ignored for bank item identity. Pack authors can extend the tag via datapack; server admins can extend the config without a datapack. Both lists are synced to clients and slave servers automatically.</details> |
| ![feature] | <details><summary>Deposit-gated components — datapack tag + config for pack authors</summary><br>A second component set drives the deposit equivalence gate: tag <code>banksystem:deposit_gated_components</code> (shipped with optional entry <code>tfc:food</code>) plus config list <code>Bank.ADDITIONAL_DEPOSIT_GATED_COMPONENTS</code>. Gated components are ignored for identity but must match a fresh withdrawal-equivalent stack at deposit time. Synced to clients and slave servers alongside the volatile set.</details> |
| ![improvement] | `ItemID.getStack()` now returns a defensive copy so callers can no longer corrupt the stored item template; a new `getStackTemplate()` raw accessor serves hot render paths |

## UI / UX

| Tag | Change |
|---:|---|
| ![bugfix] | <details><summary>Items with time-varying data (e.g. TFC food) no longer corrupt bank item identity</summary><br>Bank item identity was bit-exact on all data components and stored live item templates. Mods like TerraFirmaCraft rewrite components such as <code>tfc:food</code> creation dates in place, so banked food templates rotted inside the registry, fresh items stopped matching their bank entries, and duplicate spurious entries were created over time.<br><br>**Fix:** volatile component types (see API section) are stripped from working copies at every identity boundary — comparison, registration, storage, save, and client sync. Existing worlds migrate automatically on load: entries that now collapse to the same identity are merged into one canonical entry with a persisted alias table, so all saved references (bank balances, markets, orders) keep resolving.</details> |
| ![bugfix] | <details><summary>Withdrawn items no longer merge into mismatched stacks of the same bank item</summary><br>Withdrawing items into a container that already held a component-mismatched stack of the same bank item (e.g. fresh food withdrawn next to a spoiled stack) merged them, silently converting the withdrawn items. All withdrawal insertion paths — Bank Terminal block inventory, Bank Download Block automation, and direct-to-player-inventory drops — now use the game's own component-aware stack equality: mismatched stacks go to a separate slot, and items are never voided when the container is full.</details> |
| ![bugfix] | Bank Download Block no longer strips data components (enchantments, food state, etc.) from items it places into empty slots of the target inventory |
| ![bugfix] | <details><summary>Items loaded from a world save no longer permanently show a numeric ID instead of their name</summary><br>When an ItemID was loaded from world data before the item registry was ready, its name was cached as the bare number (e.g. "3083") and never refreshed — so server-side name consumers (command tab suggestions and similar) showed numbers instead of item names for every persisted item.<br><br>**Fix:** the numeric name is now only a provisional placeholder. The real item name is resolved immediately at load when the registry is already available, and otherwise re-resolved automatically on the next name lookup once it becomes possible (cheap, silent retry). IDs merged into a canonical entry heal to the canonical item's name.</details> |
