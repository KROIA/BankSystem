# 2.0.4 — In Development

## Security

| Tag | Change |
|---:|---|
| ![hardening] | <details><summary>World data files are now format-versioned — a newer-mod save can no longer be misread by an older build</summary><br>`ItemIDs.nbt` and `Meta_data.nbt` now carry an explicit save-format version (files written before this release have none and are treated as legacy format 1; they are converted by a one-shot re-save on first load). A file whose version is above what the running build understands — i.e. the world was saved by a <b>newer</b> BankSystem — is refused at startup <b>before anything is loaded or mutated</b>, with all saves prohibited for the session, so a temporary mod downgrade can never corrupt the newer world data. The metadata loader additionally logs which BankSystem version last saved the world on every load and warns when a downgrade is likely.</details> |
| ![bugfix] | <details><summary>Worlds corrupted by the pre-2.0.3 ItemID load-order bug are now detected and can be repaired at startup</summary><br>Worlds that ever ran a pre-2.0.3 build after the cent coins were introduced could have their `ItemIDs.nbt` overwritten with a fresh cent-shifted default mapping while all other world data (bank balances, placed blocks, StockMarket data, balance history) still referenced the old shorts — items resolved to wrong items shifted by five positions (e.g. Diorite showing up as a money note). World load now inspects the raw persisted alias entries (before any invariant filtering) and detects this corruption via two independent signals that can never both occur on a healthy world (a money-item fingerprint alias left behind by the bad merge, plus the contiguous cent-shifted money block signature), then aborts startup with a full repair report: the complete proposed short→item remap table, the entries re-appended at fresh shorts, and a mixed-epoch caveat for data written since the corruption (such balance-history rows are reported, never deleted). The admin approves via the new one-shot <code>CONFIRM_ITEMID_REPAIR</code> setting (auto-reset after use, like <code>CONFIRM_ITEMID_MERGE</code>; the reset is only persisted once the repaired load fully succeeded, so an abort can never burn the confirmation); the previous `ItemIDs.nbt` is copied aside as <code>ItemIDs.nbt.pre-repair-&lt;timestamp&gt;</code> before the repaired mapping is persisted, and an audit record (timestamp, mod version, changed-short count) is stored in `Meta_data.nbt`. Fingerprint aliases that cannot live in the runtime alias table are preserved under a dedicated <code>quarantinedAliases</code> entry in `ItemIDs.nbt`, so re-saves can never destroy the corruption evidence and detection keeps working on later startups and future mod builds. Because the short→item map is the single source of truth, restoring it makes all world data resolve correctly again — no chunk, database or StockMarket rewrite is needed.</details> |
| ![hardening] | <details><summary>Repair proposals are cross-validated before they can be offered</summary><br>The corruption repair never trusts its own detection: the proposed old and fresh mappings are re-simulated from the live registry and every piece of evidence must line up exactly (the on-disk money block must match a fresh default assignment, and every fingerprint alias must be consistent with both candidate mappings). Any inconsistency downgrades the guard to warn-and-continue — no automatic repair is offered, nothing is blocked, nothing is touched. The report explicitly explains that a symptom-free world is likely a false alarm where nothing should be restored, and that `ItemIDs.nbt` should only be restored from a world backup when actual wrong-item symptoms are present. The full report is logged at ERROR once per evidence state (an acknowledgment stored in `Meta_data.nbt` reduces later startups to a single WARN line). An applied repair also deliberately does <b>not</b> fire the ItemID merge event — a repair is not a merge, so dependent mods (e.g. StockMarket) are not tricked into consolidating markets.</details> |
| ![bugfix] | <details><summary>Default items are no longer registered before the world's ItemID file is loaded — a boot-order bug that silently damaged existing worlds on every start</summary><br>The server bank manager registered its default blacklisted, allowed and not-removable items while it was being constructed — which happened <b>before</b> `ItemIDs.nbt` was read from disk. As a result, every start of a singleplayer or master-server world first minted roughly 27 fresh ItemIDs into the still-empty registry and only then loaded the world's saved IDs on top of them. On any world whose saved ID layout differed from that fresh assignment, this silently damaged the world on its very first boot: saved entries could keep a wrong cached item name (observed as a Diorite deposit being rejected with a message claiming <i>"money200 is not allowed"</i>), and duplicate identities were silently merged with the freshly minted ID winning — renumbering saved IDs (observed as bedrock moving from its saved ID 71 to session ID 1 via a new alias) and irreversibly deleting the balance-history rows of the renumbered ID. All default item registration now happens strictly <b>after</b> the file has been loaded, so saved worlds are read into a clean registry.</details> |
| ![hardening] | <details><summary>ItemID registration is now locked until the world's ID file has been loaded</summary><br>On singleplayer and master servers, any attempt to register a new ItemID before `ItemIDs.nbt` has been loaded (or the world has been confirmed to be brand-new) is now refused with an ERROR log instead of minting an ID. This makes the entire "IDs created before the save file is read" bug class structurally impossible — including for future code changes and for third-party mods calling the public API too early. Slave servers, which receive their IDs from the master server instead of loading them from disk, are unaffected.</details> |
| ![hardening] | <details><summary>IDs loaded from disk now always win duplicate-identity merges — loading can never renumber a saved ID</summary><br>When duplicate ItemID identities are consolidated at load time, an ID that came from `ItemIDs.nbt` is now always chosen as the canonical one over any ID minted during the current session (previously the lowest number won, regardless of origin). The save file is the single source of truth: loading a world can never renumber an ID it has already persisted. Balance-history deletion during consolidation is additionally guarded so it can never touch a saved ID that lost a merge to a session-minted one.</details> |
| ![hardening] | <details><summary>Renumbering tripwire: every save now stores an ID fingerprint that is verified on the next load</summary><br>On every save, a compact ID→item-name digest is written to `Meta_data.nbt`; on every load it is compared against the live registry, and any saved ID that now resolves to a <b>different</b> item than in the last session is reported with a loud ERROR. This check is report-only — nothing is ever changed automatically — and confirmed merges or repairs stay silent because the comparison follows alias resolution. It catches entire future classes of silent renumbering bugs, not just the ones fixed in this release.</details> |
| ![hardening] | <details><summary>The corrupted-world repair now recognizes worlds that were corrupted under the real boot order of affected releases</summary><br>The repair's cross-validation now also models the actual historical boot order of the affected releases (default blacklist items registered first, then the remaining defaults), so worlds corrupted under those releases are correctly recognized instead of failing validation and being left unrepairable; corruption detection likewise checks both historical money-item orders. As before, a repair plan is only ever offered when the evidence matches exactly <b>one</b> candidate mapping — ambiguous evidence never produces a proposal.</details> |
| ![bugfix] | <details><summary>Recovering from an unreadable `ItemIDs.nbt` can no longer leave the allowed-items list permanently broken</summary><br>Recovery from an unreadable `ItemIDs.nbt` could previously end with an allowed-items list containing only a single invalid placeholder entry — making every item unbankable, forever, with no way to fix it in-game. Invalid entries are now filtered out at setup, save and load, and a restored allowed-items set that ends up empty is re-seeded with the default item list.</details> |
| ![bugfix] | <details><summary>Slave servers no longer produce a null-pointer error in the log on startup when connecting to a master</summary><br>On a slave server, the handshake with the master could complete on a background network thread before the slave finished setting up its bank manager on the main server thread — the handshake callback then tried to notify the master of the online player list through a bank manager that was briefly still null and threw a null-pointer error into the log. The startup sequence has been reordered so the bank manager and command handler are fully assigned before the master connection is opened, guaranteeing that any handshake callback observes a fully initialized state. As defense in depth, the callback also short-circuits with a clear warning if the bank manager is ever null (which can otherwise still occur in reconnect-time races between a server stop and the next server start).</details> |
| ![hardening] | <details><summary>Slave servers now wait for the master's ItemID map before registering their own default items — prevents cross-server ID divergence after a mod update adds new items</summary><br>When a mod update introduced new items to an existing master/slave setup, the slave used to mint its own local ItemID shorts for all currently-registered items on startup, <b>before</b> the master's authoritative ItemID map arrived over the network. The freshly-minted slave shorts then collided with the master's alias plan, so items added in the update (for example, the newly-shipped Bank Terminal and ATM blocks, or blocks contributed by other mods) resolved to a different bank identity on the slave than on the master — every deposit or withdraw of those items on the slave routed to a different bank account than the master would use. The slave-side registration latch shipped earlier in v2.0.4 for the master (which enforces "wait for the save file before registering defaults") now also covers slaves, treating the master's sync packet as the slave's equivalent of the save file. Default items are registered only after the sync has been applied, and the registration is register-if-absent so the master's authoritative shorts always win. If the sync never arrives (master unreachable), the slave logs a clear warning after 30 seconds so the missed sync is visible in the log instead of silently sitting with an armed latch.</details> |
| ![bugfix] | <details><summary>Fresh slave servers no longer render the wrong item names in the Bank Terminal — item lists shared between master and slave now travel by stable resource name instead of a per-startup registry slot</summary><br>When a slave server connected to a master with a pre-existing world for the first time, the Bank Terminal balance rows could render completely wrong item names — for example a "0 money" row showing up as "0 Chain commandblock". Root cause: the master→slave item-list sync packet, the slave→master item-registration packet, and the account-icon field of the account-update request all encoded each item using the sender's own runtime item slot number, which each server assigns independently at startup. Even with an identical mod set on both sides, the master's slot 4137 for <code>banksystem:money</code> could be a completely different item on the slave, so the slave stored the master's shorts against whatever item happened to sit at those slots on its own registry. Master and slave now identify each item on the wire by its stable Minecraft resource name (<code>banksystem:money</code>) instead of the per-startup slot number, so both sides always agree on which item is which regardless of load order. If the receiving side lacks the mod for an item the sender is announcing, that entry is now dropped with a single INFO log line naming the missing resource (once per item per session) instead of silently binding the short to a wrong item on the local registry — bank balances and market data for that item simply do not appear on the side that can't resolve it, and no wrong-item deposits or withdrawals are possible.</details> |
| ![feature] | <details><summary>Bank operations rejected by the trust or client-permission gate now tell the affected player why — previously the rejection was logged only on the master's console and the player saw nothing happen</summary><br>When a Bank Terminal deposit (or any other forwarded bank operation) was rejected on the master because the calling slave was not marked as trusted, or because the request was initiated directly by a client for an operation that isn't allowed from clients, the rejection used to be recorded only in the master's console log — the player who triggered the action saw no chat message, no error, and no indication that the action had failed. The master now sends a chat message to the affected player explaining the reason: for an untrusted-slave rejection, <i>"Your server '&lt;slave name&gt;' is not trusted by the bank master. This action is not allowed. Contact a bank admin to grant trust for cross-server bank operations."</i>; for a client-not-allowed rejection, <i>"This action is not allowed to be initiated directly from a client. If you triggered it through normal in-game UI, this is a bug — please report it."</i> When the master does not know which specific player on the offending slave initiated the request (server-side flows do not currently carry the initiating player's identity across the master/slave boundary), the message is broadcast to every player online on that slave — a temporary tradeoff pending a future refinement that propagates the player identity end-to-end.</details> |
| ![hardening] | <details><summary>Master is now the sole ItemID minter — slave servers never assign shorts locally, closing every remaining cross-server ID divergence class</summary><br>Slave servers no longer mint any ItemID shorts on their own. Every runtime ItemID registration on a slave is delegated to the master via a batched network round-trip; the master's response is the authoritative answer. On a cache hit the slave answers instantly from the map already synced from the master; on a miss it fires exactly one request per unknown item (subsequent lookups of the same unknown item hit a negative cache and never contact the master again), and the missing item stays unregistered until the master's response arrives. If the master doesn't recognise an item at all (a mod installed only on the slave), the response is a hard rejection: no bank identity is assigned, no bank account can carry a balance in it, and an INFO log names the item so mixed-mod setups leave a trail. Component identity is honoured — a component surviving normalisation is part of an item's identity, so an item whose component types the master's registry doesn't know is rejected outright, never silently degraded to a plain base item. Reconnecting to the master keeps the previously-known IDs and clears only the negative cache, so a mod update on the master while the slave was offline can resolve items that were previously rejected. This makes cross-server ID divergence structurally impossible even on setups where the slave and master run different mod loadouts.</details> |
| ![hardening] | <details><summary>Untrusted slave servers can no longer perform bank writes — deposit, withdraw, and every account-management operation now require slave trust</summary><br>Bank operations forwarded from a slave to the master identify the acting player by a UUID the slave supplies. A modified <b>untrusted</b> slave could forge that UUID — including a bank admin's — to satisfy the master's per-player permission checks and then deposit, withdraw, edit balances, change account settings, or rewrite user data. All write-side request handlers now reject any request that comes from a slave which is not on the master's trusted list, <b>before</b> any per-player check runs — so an untrusted slave cannot mutate bank data no matter which player UUID it claims. Deposit/withdraw return a no-op (nothing is credited or paid out) and account edits are refused; trusted slaves and direct master-side play are unaffected, and read-only requests remain available to untrusted slaves. This builds on the master's authenticated slave-identity binding, so the trust decision rests on the network connection rather than on data the slave can choose.</details> |
| ![bugfix] | <details><summary>Depositing an item the master server doesn't recognise on a slave now tells the player instead of silently doing nothing</summary><br>When a player on a slave server tried to deposit an item that was not registered on the master, two problems combined to make the action silently vanish. First, the slave's server-side registration path checked whether the item's lookup returned a null id — but the lookup actually returns a sentinel <i>invalid</i> id for unknown items, not null, so the delegation request that should have registered the unknown item with the master was skipped and the local invalid id was accepted as the answer. Second, even when the delegation ran, the master's rejection was recorded only in its own console log — the player who triggered the deposit saw no feedback at all. Both are fixed: the slave now correctly recognises "unknown item" and delegates every such registration to the master, and when the master refuses an item (for example because the mod that adds it is only installed on the slave), it sends a chat message to the affected slave explaining that the item does not exist on the master server and cannot be used with this bank system. The message reaches every player currently online on that slave — the same trade-off as the trust/client-permission rejection message shipped earlier in this release, pending a future refinement that carries the initiating player's identity across the master/slave boundary.</details> |

## API

| Tag | Change |
|---:|---|
| ![feature] | <details><summary>New lifecycle signals let dependent mods time actions against the slave&harr;master connection</summary><br>`IBankSystemEvents` gains two new signals for mods that build on BankSystem's multi-server layer: <b>slave-connection-accepted</b> fires on the slave once the handshake with the master has completed and BankSystem's cross-server accessors will genuinely round-trip to the master (issuing them earlier fast-fails while the handshake is still in flight); <b>slave-connection-lost</b> fires when the established connection to the master drops, whether unexpectedly or on a clean shutdown. Dependent mods that keep per-connection state derived from the master — for example StockMarket's cache of whether this slave is currently trusted — can now invalidate on the lost signal and refresh on the next accepted signal instead of guessing at handshake timing. Both signals follow the same idiom as the existing setup-completed and master-slave-connected signals. They fire on a network I/O thread, so listeners must return quickly and hand off any real work to their own thread.</details> |
| ![feature] | <details><summary>New master-side event fires when the slave-trust set changes so dependent mods can propagate the flip live</summary><br>`IBankSystemEvents` gains a <b>trusted-slaves-changed</b> event that fires on the master server every time <code>/banksystem trust &lt;slave&gt;</code> or <code>/banksystem untrust &lt;slave&gt;</code> (or any other caller of the trust-set mutators) writes the master's slave-trust set. The payload carries the affected slaveID and the post-mutation trusted state, so subscribers can propagate the new value directly without re-querying the master. Dependent mods (e.g. StockMarket) can subscribe to push mid-session trust-toggle updates to their own connected slaves and downstream clients — no polling, and no waiting for the next reconnect handshake. Fires on the <b>master JVM only</b> and never on slave JVMs, so the master's cross-slave trust set is not leaked to individual slaves; slaves learn about their own trust status through S2S packets dispatched by mods that subscribe here on the master. Fires on the server main thread AFTER the trust set has been mutated, so listeners observe the post-change state.</details> |
| ![feature] | <details><summary>New master-side event fires when a slave disconnects, so dependent mods can release per-slave state</summary><br>`IBankSystemEvents` gains a <b>master-slave-disconnected</b> event — the counterpart to the existing master-slave-connected signal — that fires on the master when a connected slave drops (unexpected loss or clean shutdown). Its payload carries the departing slave's ID so subscribers can evict any per-slave state they built while it was connected; the master previously fired nothing at all when a slave left. Note the deliberate asymmetry with the connected signal, which is parameterless: the disconnected event carries the slaveID because per-slave eviction needs to know which slave left. Fires on the master JVM only.</details> |
| ![hardening] | <details><summary>The slave-connection-lost signal now fires exactly once per disconnect instead of repeating while the master is offline</summary><br>Previously the slave's automatic reconnect loop re-fired the <b>slave-connection-lost</b> signal on every failed reconnect attempt — roughly every five seconds for the entire duration of a master outage — so a dependent mod's cache-invalidation handler could be re-run indefinitely. The signal is now edge-triggered: it fires once on the connected&rarr;disconnected transition and stays silent through the reconnect-retry storm, re-arming only after the next successful handshake fires slave-connection-accepted again. Listeners no longer need to be idempotent to avoid repeated work.</details> |
| ![feature] | <details><summary>Bank-account selection can now filter by <i>all</i> requested permissions (AND), not only <i>any</i> (OR)</summary><br>`BankAccountData` gains `hasAllPermissions(UUID, int)` — the AND-semantics counterpart to the existing `hasAnyPermission` (returns true only when the user holds every bit in the mask) — and `BankAccountSelectionScreen` gains a constructor overload with a <code>requireAllPermissions</code> flag. When set, the picker lists only accounts where the player holds <b>every</b> requested permission (e.g. both DEPOSIT and WITHDRAW); the existing constructors keep the previous any-permission (OR) behavior unchanged. This lets dependent mods present a precise account picker without reimplementing the screen.</details> |

| ![bugfix] | <details><summary>A shared bank account can no longer be accidentally orphaned by removing its last managing user — there is always at least one user with Manage permission</summary><br>On a shared (non-personal) bank account, the account-management window let a manager remove every user — including themselves as the last one — which left the account with nobody able to manage it and, because an account with no users was then deleted, permanently lost its balance. The notification now fires from the deposit action itself: it appears on <b>every</b> deposit attempt, names the <b>specific</b> item, and is sent only to the <b>player who made the deposit</b>. Bulk/passive ID-registration traffic no longer produces spurious "you tried to bank…" messages.</details> |

Settings that are read once at startup carry a clearly visible "⟳ Restart required" marker with a tooltip explaining that the change is saved now but takes effect after a restart.
